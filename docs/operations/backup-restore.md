# Backup and Restore

The hive team server stores all shared sessions in a single SQLite file. Back it up regularly to protect your team's collective AI coding history.

## Which database to back up

| Database | Location | Priority |
|---|---|---|
| `server.db` | `~/.local/share/hive/server.db` | **High** -- team-wide shared history |
| `store.db` | `~/.local/share/hive/store.db` | Low -- local client, can be rebuilt from JSONL transcripts |

In team mode, `server.db` is the important one. Each developer's `store.db` can be regenerated by running `hive init` (which backfills from Claude Code transcripts).

## Manual backup

Use SQLite's built-in `.backup` command for a safe, consistent snapshot:

```bash
sqlite3 ~/.local/share/hive/server.db \
    ".backup /backups/server-$(date +%Y%m%d).db"
```

!!! tip "Hot backup"
    The `.backup` command works while the server is running. It uses SQLite's backup API to produce a consistent copy without stopping writes.

## Automated daily backups

Add a cron job that runs the backup daily and retains 30 days of history:

```bash title="crontab -e"
0 3 * * * sqlite3 /home/hive/.local/share/hive/server.db ".backup /backups/hive/server-$(date +\%Y\%m\%d).db" && find /backups/hive/ -name "server-*.db" -mtime +30 -delete
```

This runs at 3:00 AM, creates a dated backup, and deletes backups older than 30 days.

For a more readable setup, use a script:

```bash title="/usr/local/bin/hive-backup.sh"
#!/bin/bash
set -euo pipefail

BACKUP_DIR="/backups/hive"
DB_PATH="/home/hive/.local/share/hive/server.db"
RETAIN_DAYS=30

mkdir -p "$BACKUP_DIR"
sqlite3 "$DB_PATH" ".backup $BACKUP_DIR/server-$(date +%Y%m%d).db"
find "$BACKUP_DIR" -name "server-*.db" -mtime +$RETAIN_DAYS -delete

echo "Backup complete: $BACKUP_DIR/server-$(date +%Y%m%d).db"
```

```bash
chmod +x /usr/local/bin/hive-backup.sh
```

```bash title="crontab -e"
0 3 * * * /usr/local/bin/hive-backup.sh >> /var/log/hive-backup.log 2>&1
```

## Restoring from backup

### 1. Stop the server

```bash
sudo systemctl stop hive
```

### 2. Replace the database

```bash
cp /backups/hive/server-20260419.db \
   /home/hive/.local/share/hive/server.db
chown hive:hive /home/hive/.local/share/hive/server.db
```

!!! warning "Remove WAL files"
    Delete any stale WAL files before starting the server with a restored database:
    ```bash
    rm -f /home/hive/.local/share/hive/server.db-wal
    rm -f /home/hive/.local/share/hive/server.db-shm
    ```

### 3. Restart the server

```bash
sudo systemctl start hive
curl -s http://localhost:3000/ | jq .status
```

```
"ok"
```

## Restoring local `store.db`

If a developer loses their local database, they do not need a backup. Run:

```bash
hive init
```

This recreates the database and backfills sessions from Claude Code's JSONL transcripts in `~/.claude/projects/`. The transcripts are the source of truth for local sessions.

## Offsite backup

For critical deployments, copy backups offsite:

```bash
# Sync to S3
aws s3 sync /backups/hive/ s3://my-bucket/hive-backups/ --storage-class STANDARD_IA

# Or rsync to another machine
rsync -az /backups/hive/ backup-host:/backups/hive/
```

## Verifying backups

Periodically check that backup files are valid:

```bash
sqlite3 /backups/hive/server-20260419.db "SELECT count(*) FROM sessions;"
```

If this returns a number without errors, the backup is intact.
